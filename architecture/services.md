# Homelab Services Architecture

**Environment:** Ubuntu Server 24.04 LTS \
**Identity:** Samba 4 Active Directory DC (bare-metal — provisioned before Docker) \
**Docker Networks:** `proxy`, `databases` \
**Base Domain:** `*.in.alybadawy.com`

---

## Identity — Samba 4 Active Directory (Bare-Metal)

Samba AD is **not** a Docker service. It runs directly on the host OS and must be set up before Docker is installed.

| Component     | Details                                                             |
| ------------- | ------------------------------------------------------------------- |
| Service name  | `samba-ad-dc` (systemd)                                             |
| Realm         | `ID.IN.ALYBADAWY.COM`                                               |
| Domain        | `id.in.alybadawy.com`                                               |
| NetBIOS name  | `ID`                                                                |
| DC hostname   | `dc.id.in.alybadawy.com` → `172.20.20.5`                            |
| DNS backend   | Samba Internal (port 53 on host)                                    |
| DNS forwarder | `172.20.20.1` (UDR7)                                                |
| LDAP          | Port 389 (TCP) — LAN accessible                                     |
| LDAPS         | Port 636 (TCP) — LAN accessible                                     |
| Kerberos      | Port 88 (TCP/UDP) — LAN accessible                                  |
| Config        | Generated by `samba-tool domain provision` at `/etc/samba/smb.conf` |

**Services that authenticate via Samba AD:**

| Service    | Method | Notes                                                   |
| ---------- | ------ | ------------------------------------------------------- |
| Nextcloud  | LDAPS  | Uses LDAP User and Group Backend app; service account   |
| NAS        | LDAPS  | Bind with service account; maps AD groups to NAS shares |
| Immich     | OIDC   | OIDC provider to be determined (may proxy via AD)       |
| Home Asst. | OAuth2 | OAuth2 provider to be determined                        |

See the rebuild guide: [`rebuild/active-directory/`](../rebuild/active-directory/)

## Core Infrastructure

| Service      | Image                           | Networks     | Subdomain / Access                |
| ------------ | ------------------------------- | ------------ | --------------------------------- |
| Portainer    | portainer/portainer-ce:latest   | proxy        | `dockers.in.alybadawy.com`        |
| NPM          | jc21/nginx-proxy-manager:latest | proxy        | `proxy.in.alybadawy.com`          |
| Beszel Hub   | henrygd/beszel                  | proxy        | `monitor.in.alybadawy.com`        |
| Beszel Agent | henrygd/beszel-agent            | host network | _(connects to hub at port 45876)_ |

**Notes:**

> [!TIP]
>
> - Portainer is the primary orchestration interface — all stacks after the initial bootstrap are deployed and managed through it
> - NPM handles TLS termination and reverse proxying; wildcard cert deployed by acme.sh (bare-metal) to `/opt/certs/`, mounted read-only into NPM
> - Beszel Hub collects metrics from Beszel Agent; agent uses `network_mode: host` to observe real host-level CPU, memory, disk, and network stats
> - acme.sh runs bare-metal (not Docker) and renews certs via a host cron job — see ADR-006

## Database Layer

| Service             | Image          | Network   | Purpose          | Notes                                                                                                 |
| ------------------- | -------------- | --------- | ---------------- | ----------------------------------------------------------------------------------------------------- |
| PostgreSQL (Global) | postgres:16    | databases | Multi-tenant DB  | Single instance serving all apps — Nextcloud, Immich, and others; each app uses its own database name |
| Redis (Global)      | redis:7-alpine | databases | Cache & sessions | Shared instance used by Nextcloud, Immich, and others                                                 |

**Notes:**

> [!TIP]
>
> - Apps connect to PostgreSQL using the `postgres` superuser — each app creates its own database on first run
> - Apps on the `databases` network can reach both containers by hostname (`postgres`, `redis`)

## Applications

| Service                 | Image                                              | Networks           | Subdomain / Access        |
| ----------------------- | -------------------------------------------------- | ------------------ | ------------------------- |
| Nextcloud               | lscr.io/linuxserver/nextcloud:latest               | proxy, databases   | `cloud.in.alybadawy.com`  |
| Immich Server           | ghcr.io/immich-app/immich-server:release           | proxy, databases   | `photos.in.alybadawy.com` |
| Immich Machine Learning | ghcr.io/immich-app/immich-machine-learning:release | _(stack-internal)_ | _(no external access)_    |
| Home Assistant          | ghcr.io/home-assistant/home-assistant:stable       | host network       | `ha.in.alybadawy.com`     |

**Notes:**

> [!TIP]
>
> - Nextcloud and Immich both use the global PostgreSQL and Redis via the `databases` network
> - Immich Machine Learning communicates with Immich Server within the stack; it needs no external network access
> - Home Assistant uses `network_mode: host` for mDNS and device discovery; NPM proxies it via `127.0.0.1:8123`
> - NFS mounts at `/mnt/nas/homelab/{nextcloud,immich}` must be present on the host before these stacks are deployed

## Secrets Management

All secrets are stored as **environment variables in Portainer** — defined per-stack in the **Environment variables** section of the Portainer stack editor. No `.env` files on disk.

Key secrets to record in your password manager:

| Secret                     | Used by                           |
| -------------------------- | --------------------------------- |
| `POSTGRES_PASSWORD`        | All apps connecting to PostgreSQL |
| `NEXTCLOUD_ADMIN_PASSWORD` | Nextcloud initial setup           |
| `BESZEL_KEY`               | Beszel agent authentication       |
| `BESZEL_TOKEN`             | Beszel agent authentication       |

## Deployment Order

Matches the rebuild guide sequence:

1. OS Installation
2. Active Directory (Samba 4) — bare-metal, before Docker
3. NAS Mounting — NFS mounts must be present before Docker services that use them
4. Docker + Portainer — container runtime; Portainer bootstrapped from CLI
5. SSL Certificates — acme.sh + Vercel DNS wildcard cert to `/opt/certs/`
6. Core Infrastructure — NPM deployed via Portainer
7. Databases — PostgreSQL and Redis deployed via Portainer
8. Monitoring — Beszel hub + agent deployed via Portainer
9. Nextcloud
10. Immich
11. Home Assistant

Samba AD **must** be healthy before any Docker services start, because app containers connect to it for authentication on first run.

## Monitoring & Health Checks

- **Portainer Dashboard:** Container status, resource usage, logs, stack management
- **Beszel Dashboard (`monitor.in.alybadawy.com`):** Real-time host metrics (CPU, RAM, disk, network) and per-container stats
- **Docker health checks:** `healthcheck:` blocks defined in PostgreSQL and Redis stacks; check status with `docker ps`

Recommended container restart policy: `restart: unless-stopped` for all production stacks.
